# 6조

# 1. 배경

기존의 수기와 같은 오프라인 방식이 가진 한계(ex. 실시간 재고 관리가 안됨, 반납 지연시 추적이 어려움)를 온라인화를 통해서 해결하도록한다.

사용자들은 현재 대여 상황과 재고 상황을 확인할 수 있다.

관리자들은 관리가 어려운 종이 형식의 수기가 아닌 온라인 데이터를 통해서 관리 및 승인을 더 간편하게 할 수 있다.

# 2. 목표

오프라인으로는 재고 관리가 어려운 반면 온라인을 통해 재고 확인이 가능한 사용자 경험을 개선한 사용자 친화적 관리 시스템을 구축

사용자 친화적 UI를 통해 UX 개선

반납 예정일 알림, 연체시 알람을 통한 미반납율 10% 감소

# 3. 페르소나

센트럴파크에서 신입생이 돗자리를 대여하여 피크닉을 하고 싶을 때, 서비스를 통해 재고를 확인하고 돗자리를 예약한다.

학생이 재고를 확인 후 대여를 신청하면 관리자는 예약을 승인한 뒤, 학생이 예약된 시간에 물품을 대여하러 오면 관리자는 제품을 대여해주고 예약 승인 상태를 대여 상태로 바꾼다.

물품을 대여한 학생이 반납을 잊어버리고 있는 상태에서 푸시 알람을 통해서 반납 시간을 확인하게 해서 반납률을 상승시킨다.

# 4. 핵심 기능

1. 로그인/로그아웃, 프로필
    - 학교 계정(knu.ac.kr)을 통해서 로그인
    - 프로필을 통해 사용자 정보 확인
2. 재고 확인(대여 상태)
    - 4단계(Available → Reserved → Rented → Returned/Overdue) 프로세스를 통해 재고 상태 확인
3. 대여 기록
    - 대여했던 물품 목록 확인
4. 동시성 보장
    - 동시 예약시 데이터 정합성 보장
5. 푸시 알람
    - 예약일 고지
    - 만료일, 시간 고지
    - 연체시 알림

# 5. 제외 범위

초반 개발에서는 푸시 알람을 제외한다.

웹 플랫폼으로 개발 목표, pwa는 웹 개발 이후 추진을 목표로 한다.

# 6. 성공 지표

반납률 10프로 상승

대여율 15퍼 상승

관리자가 이전보다 물품의 추적이 용이해진다. (사용자 친화적)

# 7. 예외 케이스

학교 졸업, 휴학, 자퇴 학생 관리를 어떻게 할 것인가?

예약 동시성을 어떻게 처리할 것인가?

# 주차별 계획

## FE

### **1주차: 기획 분석, UX 전략 수립 및 초기 세팅**

- **PRD 및 디자인 리뷰 (UX 포인트 파악):**
    - 유저가 서비스를 이용하는 핵심 목적(User Journey)을 파악하고 페인 포인트(Pain Point) 해결을 위한 UX 전략 수립.
    - 디자인 시스템(컬러, 타이포그래피, 레이아웃) 검토.
- **프로젝트 초기 세팅:**
    - 사용할 프레임워크(React, Next.js 등) 및 라이브러리(스타일링, 상태관리) 세팅.
    - 폴더 구조 설계 및 ESLint/Prettier 등 코드 컨벤션 확립.
- **핵심 화면 Flow 설계:**
    - 전체 페이지 라우팅 구조 설계.
    - 권한에 따른 접근 제어(로그인 유저 vs 비로그인 유저) Flow 기획.

### **2주차: 레이아웃 구현 및 핵심 화면 Flow 연결**

- **공통 UI 컴포넌트 개발:**
    - 버튼, 모달, 입력 폼(Form), 내비게이션 바 등 재사용성이 높은 UI부터 우선 개발하여 작업 속도 향상.
- **핵심 화면 마크업 및 Flow 구현:**
    - 메인 페이지, 주요 서비스 진입 페이지 등 코어 화면의 레이아웃 완성.
    - 페이지 간 기본 화면 전환(Routing)이 정상적으로 이루어지는지 Flow 연결.
- **기본 UX 요소 적용:**
    - 로딩 상태를 보여주는 Skeleton UI 또는 Spinner 뼈대 작업.
    - 잘못된 접근을 막는 에러 페이지(404) 구현.

### **3주차: 상세 비즈니스 로직, API 연동 및 이미지 전략 적용**

- **API 연동 및 동적 데이터 바인딩:**
    - 백엔드 API를 연결하여 화면에 실제 데이터 렌더링.
    - 폼(Form) 제출, 유효성 검사 및 에러 메시지 노출 등 세부 인터랙션 구현.
- **이미지 및 미디어 전략 적용:**
    - **로딩 최적화:** 초기 로딩 속도 개선을 위한 Lazy Loading(지연 로딩) 적용.
    - **반응형 대응:** 디바이스 크기에 맞는 이미지를 불러오도록 `srcset` 적용.
    - **UX 향상:** 이미지가 로드되기 전 공간이 무너지는 것을 막기 위한 Placeholder(블러 처리된 저화질 이미지 등) 적용.
    - **포맷 최적화:** WebP 등 차세대 이미지 포맷을 지원하도록 구성.

### **4주차: 상태 고도화, QA 및 배포 준비**

- **상태 관리 및 성능 최적화:**
    - 전역 상태 관리 고도화 및 불필요한 리렌더링(Re-rendering) 방지.
    - API 중복 호출 방지를 위한 캐싱 및 디바운스/쓰로틀링 적용.
- **크로스 브라우징 및 반응형 QA:**
    - 모바일/태블릿/데스크탑 환경에서 UI가 깨지지 않는지 검수.
    - 주요 브라우저(Chrome, Safari 등) 환경 호환성 테스트 및 버그 수정.
- **최종 디버깅 및 배포:**
    - 테스트 과정에서 발견된 이슈 픽스.
    - 프로덕션 빌드 후 Vercel, AWS 등을 활용한 실제 환경 배포.

## 🗓️ 4주 MVP 백엔드 개발 로드맵: "빌려줄게(가칭)"

> **목표:** 4주 안에 안정적인 대여/예약 시스템의 핵심 로직 완성 (웹 기준, 푸시 제외)
> 

### 🏗️ Week 1: 뼈대 구축 및 인증 (Foundation)

**"서비스의 시작점과 보안을 잡는 시기"**

- **설계:** 아키텍처(Layered) 확정 및 **ERD 작성** (User, Item, Rental, Reservation)
- **상태 정의:** `AVAILABLE`, `RESERVED`, `RENTED`, `RETURNED`, `OVERDUE` 등 Enum 세팅
- **인증:** 학교 메일(`@knu.ac.kr`) 도메인 검증 및 **JWT 기반 Auth** 환경 구축
- **기본 세팅:** Spring Boot/Node.js 프로젝트 초기화, 공통 응답 및 예외 처리(Global Exception Handler) 구성

---

### 📦 Week 2: 핵심 비즈니스 로직 (Core Feature)

**"사용자가 물건을 보고 예약할 수 있게 만드는 시기"**

- **물품/재고:** 물품 리스트 조회 API (상태별 필터링 포함)
- **예약 시스템:** 예약 생성 및 취소 로직 구현
- **상태 전이(State Transition):**
    - `관리자 승인` → **RESERVED**
    - `대여 시작` → **RENTED**
    - `반납 완료` → **RETURNED**
- **마이페이지:** 사용자별 대여/예약 히스토리 조회 API

---

### 🔒 Week 3: 데이터 정합성 및 관리자 (Reliability)

**"동시에 몰려도 터지지 않는 신뢰도를 쌓는 시기"**

- **동시성 제어 (Critical!):** * 동일 물품에 대한 중복 예약 방지 로직
    - **Pessimistic Lock**(DB 비관적 락) 또는 Redis 분산 락 적용
- **관리자 전용:** 예약 승인, 상태 수동 변경, 반납 확인 API
- **배치 처리:** 반납 기한 경과 시 상태를 `OVERDUE`로 자동 전환하는 로직 (or 조회 시 계산)
- **검증:** 이미 대여 중인 물품에 대한 예약 방지 등 엣지 케이스 방어

---

### 🛠️ Week 4: 안정화 및 고도화 (Final Polish)

**"실제 서비스 수준으로 품질을 끌어올리는 시기"**

- **최적화:** JPA N+1 문제 해결 및 인덱스(Index) 설정을 통한 조회 성능 개선
- **문서화:** **Swagger(OpenAPI)**를 이용한 API 명세서 자동화 및 정리
- **테스트:** JUnit 등을 활용한 **핵심 로직(예약/동시성/상태 변경)** 단위 테스트
- **모니터링:** 주요 비즈니스 이벤트(대여/반납/에러) 로그 기록 및 정리
